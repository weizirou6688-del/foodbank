# AI 端到端检查与排错短句清单

适用范围：本地开发、前后端联调、页面验活、接口核对、数据口径对齐、旧进程清理。

文档目标：把高频错误点、检查方法、排错角度、常用短语整理成可直接复用的句子和清单。

## 1. 一句话原则

- 先验活，再猜原因。
- 先查进程，再查代码。
- 先看端口，再看页面。
- 先打接口，再看 UI。
- 先确认是不是旧进程。
- 先确认是不是同一批服务。
- 先确认是不是本项目进程。
- 先确认是不是 live 数据。
- 先确认字段名是否改全。
- 先确认口径是否一致。
- 先做干净重启，再继续深挖。
- 先缩小变量，再扩大排查范围。

## 2. AI 做细端到端检查时最常发现错误的地方

### 2.1 进程与端口

- 旧进程还在。
- 端口还在监听，但不是本项目服务。
- 前端和后端不是同一轮启动出来的。
- `node` 进程还活着，但页面已经换了版本。
- `python` 进程还活着，但数据库连接已经失效。
- 父进程退出了，子进程还在占端口。
- `.logs\frontend.port` 和真实监听端口不一致。
- `.logs\backend.port` 和真实监听端口不一致。

### 2.2 启动与环境

- `.venv` 和系统 Python 混用。
- `dev.env` 的端口配置和实际运行端口不一致。
- 前端起来了，但代理目标不是当前后端。
- 后端起来了，但数据库并没有真正可用。
- 启动脚本成功了，但实际页面吃的是旧缓存或旧实例。
- 依赖装好了，但运行时环境变量不完整。

### 2.3 页面与接口

- 页面能打开，但数据是静态占位。
- 页面能打开，但接口没有真正请求成功。
- 页面接的是旧字段，后端回的是新字段。
- 页面标签改了，`key` 没改。
- 页面调用的是新接口，组件映射还按旧结构写。
- 页面看起来正常，实际读的是 mock 数据。
- 参考 HTML 在，注入桥接不在。

### 2.4 数据与口径

- 捐赠页和仪表板统计口径不一致。
- “本月” 和 “全年” 被混算。
- “全部时间” 和 “当前区间” 被混算。
- `pending` 和 `received` 被混算。
- 家庭数按申请条数算了，没有按家庭去重。
- 机构数按捐赠记录条数算了，没有按唯一 donor 去重。
- 同一指标在不同页面使用了不同的过滤条件。
- 同一个接口改了算法，另一个接口没有同步改。

### 2.5 认证与权限

- 登录成功了，但角色不对。
- 页面是 admin 页，接口走的是 public 路由。
- token 还在，但已经来自旧环境。
- token 还在，但权限不够。
- 页面能访问，局部数据接口被权限挡住。

### 2.6 构建与运行

- 单元测试过了，真实页面还是报错。
- `npm run build` 过了，运行时注入链还是断的。
- 后端测试过了，真实数据库数据还是不对。
- 类型定义改了，页面映射没跟上。
- 接口返回正常，组件状态没有更新。

## 3. 可以从什么角度排错

- 进程角度。先问是不是旧进程。
- 端口角度。先问这个端口到底是谁在监听。
- 健康检查角度。先问服务是不是活着，数据库是不是通着。
- 页面入口角度。先问用户实际打开的是哪个路由。
- 接口角度。先问请求有没有真的打出去。
- 鉴权角度。先问这个接口是否需要登录或特定角色。
- 注入链角度。先问页面拿到的是 live service 还是占位数据。
- 字段映射角度。先问后端返回字段和前端读取字段是否一致。
- 数据口径角度。先问两个页面是不是在算同一个定义。
- 日志角度。先看 `.logs`，再看源码。
- 脏工作区角度。先分清本轮改动和历史遗留改动。
- 最小复现角度。先验证单个接口，再验证整页流程。

## 4. 推荐的端到端检查顺序

1. 先停掉旧服务。
2. 再启动一套干净服务。
3. 再确认前后端最终端口。
4. 再确认监听这些端口的 PID 和命令行。
5. 再打后端健康检查。
6. 再打前端首页探活。
7. 再打关键公共接口。
8. 再打关键鉴权接口。
9. 再打开目标页面验证 live 数据。
10. 再对比跨页面同一指标。
11. 再跑相关测试。
12. 再跑构建。

## 5. 旧进程清理的检查方法

- 最稳的做法是先跑 `scripts\stop.bat`。
- 不要一上来就把所有 `node` 和 `python` 全部杀掉。
- 先确认端口，再确认 PID，再确认命令行。
- 只清理属于本项目的进程。
- 清理后要再次确认端口是否真的释放。
- 重启后要再次确认是不是新 PID。

### 常用动作

- 停服务：`scripts\stop.bat`
- 启服务：`scripts\quick_start.bat`
- 看端口记录：`Get-Content .logs\backend.port, .logs\frontend.port`
- 看监听端口：`Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -in @(5173,8000) }`
- 看 PID 对应命令：`Get-CimInstance Win32_Process | Where-Object { $_.ProcessId -in @(PID1,PID2) } | Select-Object ProcessId,ParentProcessId,Name,CommandLine`

## 6. 本项目最实用的快速检查命令

### 6.1 服务验活

```powershell
curl.exe -fsS http://127.0.0.1:8000/health
curl.exe -I -s http://127.0.0.1:5173
```

### 6.2 端口与进程

```powershell
Get-Content .logs\backend.port, .logs\frontend.port

Get-NetTCPConnection -State Listen |
  Where-Object { $_.LocalPort -in @(5173,8000) } |
  Select-Object LocalAddress,LocalPort,OwningProcess,State
```

```powershell
Get-CimInstance Win32_Process |
  Where-Object { $_.ProcessId -in @(23132,31984) } |
  Select-Object ProcessId,ParentProcessId,Name,CommandLine
```

说明：最后一个命令里的 PID 需要替换成你当前机器上查到的值。

### 6.3 关键公共接口

```powershell
curl.exe -fsS "http://127.0.0.1:8000/api/v1/stats/public-goods-impact?range=month"
```

### 6.4 关键鉴权接口

```powershell
$body = @{ email = 'admin@foodbank.com'; password = 'admin123' } | ConvertTo-Json
$auth = Invoke-RestMethod -Uri 'http://127.0.0.1:8000/api/v1/auth/login' -Method Post -ContentType 'application/json' -Body $body
$headers = @{ Authorization = "Bearer $($auth.access_token)" }
Invoke-RestMethod -Uri 'http://127.0.0.1:8000/api/v1/stats/dashboard?range=month' -Headers $headers | ConvertTo-Json -Depth 8
```

## 7. 针对本项目的高频联调检查点

- 先看 `README.md` 的启动和停止说明是否和当前操作一致。
- 先看 `scripts/quick_start.bat` 是否已经打印真实端口。
- 先看 `scripts/stop.bat` 是否已经清掉旧监听。
- 先看 `scripts/start_backend.ps1` 和 `scripts/start_frontend.ps1` 的日志输出位置。
- 先看 `scripts/Data_dashboard.html` 是否依赖 `window.dashboardDataService`。
- 先看 `frontend/src/pages/Admin/AdminDataDashboardPreview.tsx` 是否真的注入了 `dashboardDataService`。
- 先看 `frontend/src/pages/DonateGoods/DonateGoods.tsx` 是否调用了 `getPublicGoodsImpact`。
- 先看 `frontend/src/shared/lib/api.ts` 是否已经有对应 API 封装。
- 先看 `backend/app/routers/stats.py` 是否返回前端真正使用的 `key`。
- 先看 `backend/app/services/impact_metrics_service.py` 是否被多个入口复用。

## 8. 症状到问题的快速判断

- 页面能打开，但数值全是 `--`。先查接口是否成功，再查前端映射。
- 页面能打开，但提示 live data unavailable。先查接口失败，再查注入链失败。
- Dashboard 有数据，Donate 页面没数据。先查公共接口是否返回，再查捐赠页 `useEffect` 和字段映射。
- Donate 页面有数据，Dashboard 没数据。先查 admin 登录、token、权限和 dashboard 注入链。
- 两边都有数据，但值不同。先查统计口径是否共用同一服务。
- 5173 正常，8000 不通。先查后端进程、数据库、健康接口。
- 8000 正常，5173 不通。先查 Vite 进程、代理和端口冲突。
- `build` 通过，但页面仍错。先查运行时依赖、iframe 注入、权限和真实数据。
- 测试通过，但真实数据不对。先查数据库内容、演示数据和过滤条件。

## 9. 一页式最小检查清单

- [ ] 旧进程是否已清理。
- [ ] 前后端是否已干净重启。
- [ ] `.logs\backend.port` 是否存在且可信。
- [ ] `.logs\frontend.port` 是否存在且可信。
- [ ] 8000 是否返回健康检查成功。
- [ ] 5173 是否返回 200。
- [ ] 监听 8000 和 5173 的 PID 是否属于本项目。
- [ ] 目标页面是否真的读取 live 数据。
- [ ] 关键接口是否直接返回预期字段。
- [ ] 前后端字段名是否完全对齐。
- [ ] 同一指标在多个页面的口径是否一致。
- [ ] 当前 token 和角色是否正确。
- [ ] `.logs` 是否有第一现场错误信息。
- [ ] 相关测试是否跑过。
- [ ] 构建是否跑过。

## 10. 可直接复用的排错短语

- 先确认是不是旧进程问题。
- 先确认端口占用的是不是本项目。
- 先确认前后端是不是同一轮启动的。
- 先确认页面拿到的是 live 数据，不是占位数据。
- 先确认这个错误发生在启动层、接口层，还是渲染层。
- 先确认后端字段已经改完，前端映射有没有跟上。
- 先确认两个页面看的是不是同一个时间范围。
- 先确认两个页面对同一个指标用的是不是同一个定义。
- 先确认日志里有没有第一现场。
- 先确认是不是权限问题，而不是接口不存在。
- 先确认是不是 token 过期，而不是页面逻辑错误。
- 先确认是不是 mock 和真实接口混用了。
- 先确认是不是缓存了旧页面。
- 先确认是不是 dirty worktree 里的旧改动在干扰判断。
- 先做干净重启，再做细排查。

## 11. 推荐给 AI 的排错提示词模板

### 11.1 端到端检查模板

```text
请按“进程 -> 端口 -> 健康检查 -> 关键接口 -> 页面数据 -> 跨页面口径 -> 测试/构建”的顺序做一次本地端到端检查。
先不要改代码。
先确认旧进程是否存在。
先确认 5173 和 8000 的真实监听者。
先确认前后端是不是同一轮启动的。
然后再判断问题在启动层、接口层、权限层、映射层还是数据口径层。
```

### 11.2 旧进程清理模板

```text
请先确认哪些 node/python 进程属于当前仓库。
不要直接结束所有同名进程。
优先使用仓库自带的 stop 脚本。
清理完成后，重新启动服务，并验证新的端口、PID 和健康检查结果。
```

### 11.3 数据口径对齐模板

```text
请检查页面 A 和页面 B 的同名指标是否来自同一套统计逻辑。
重点检查：
1. 是否使用同一个 service
2. 是否使用同一个时间范围
3. 是否使用相同的状态过滤条件
4. 是否使用相同的去重规则
5. 前端读取的 key 是否一致
```

## 12. 这类问题最适合的排错顺序

- 先排“服务没起来”。
- 再排“服务起来了但不是这一套”。
- 再排“接口通了但页面没接上”。
- 再排“页面接上了但字段不对”。
- 再排“字段对了但口径不一致”。
- 最后才排“算法和业务定义是不是需要重构”。

## 13. 适合作为结论句的表达

- 本轮问题首先不是代码逻辑错误，而是旧进程和端口状态混乱。
- 本轮问题首先不是接口不存在，而是页面没有吃到 live 数据。
- 本轮问题首先不是页面渲染异常，而是统计口径没有完全对齐。
- 本轮问题首先不是启动失败，而是前后端来自不同轮次的运行实例。
- 本轮问题首先不是 build 失败，而是运行时注入链断裂。
- 本轮问题首先不是数据为空，而是过滤条件或字段映射不一致。
