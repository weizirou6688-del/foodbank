# 端到端检查与排错清单

适用范围：本地开发、前后端联调、页面验活、接口核对、数据口径对齐、旧进程清理。

文档目标：把高频错误点、检查方法、排错角度和常用判断句整理成可直接复用的清单。

## 1. 一句话原则

- 先验活，再猜原因。
- 先查进程，再查代码。
- 先看端口，再看页面。
- 先打接口，再看 UI。
- 先确认是不是旧进程。
- 先确认是不是同一批服务。
- 先确认是不是本项目进程。
- 先确认是不是 live 数据。
- 先确认字段名是否改全。
- 先确认口径是否一致。
- 先做干净重启，再继续深挖。
- 先缩小变量，再扩大排查范围。

## 2. 最常见的问题来源

### 2.1 进程与端口

- 旧进程还在。
- 端口还在监听，但不是本项目服务。
- 前端和后端不是同一轮启动出来的。
- `node` 进程还活着，但页面已经换了版本。
- `python` 进程还活着，但数据库连接已经失效。
- 父进程退出了，子进程还在占端口。
- `.logs\frontend.port` 和真实监听端口不一致。
- `.logs\backend.port` 和真实监听端口不一致。

### 2.2 启动与环境

- `.venv` 和系统 Python 混用。
- `dev.env` 的端口配置和实际运行端口不一致。
- 前端起来了，但代理目标不是当前后端。
- 后端起来了，但数据库并没有真正可用。
- 启动脚本成功了，但实际页面吃的是旧缓存或旧实例。
- 依赖装好了，但运行时环境变量不完整。

### 2.3 页面与接口

- 页面能打开，但数据是静态占位。
- 页面能打开，但接口没有真正请求成功。
- 页面接的是旧字段，后端回的是新字段。
- 页面标签改了，`key` 没改。
- 页面调用的是新接口，组件映射还按旧结构写。
- 页面看起来正常，实际读的是 mock 数据。
- 参考 HTML 在，注入桥接不在。

### 2.4 数据与口径

- 捐赠页和仪表板统计口径不一致。
- “本月” 和 “全年” 被混算。
- “全部时间” 和 “当前区间” 被混算。
- `pending` 和 `received` 被混算。
- 家庭数按申请条数算了，没有按家庭去重。
- 机构数按捐赠记录条数算了，没有按唯一 donor 去重。
- 同一指标在不同页面使用了不同的过滤条件。
- 同一个接口改了算法，另一个接口没有同步改。

### 2.5 认证与权限

- 登录成功了，但角色不对。
- 页面是 admin 页，接口走的是 public 路由。
- token 还在，但已经来自旧环境。
- token 还在，但权限不够。
- 页面能访问，局部数据接口被权限挡住。

### 2.6 构建与运行

- 单元测试过了，真实页面还是报错。
- `npm run build` 过了，运行时注入链还是断的。
- 后端测试过了，真实数据库数据还是不对。
- 类型定义改了，页面映射没跟上。
- 接口返回正常，组件状态没有更新。

## 3. 推荐的检查顺序

1. 停掉旧服务。
2. 启动一套干净服务。
3. 确认前后端最终端口。
4. 确认监听这些端口的 PID 和命令行。
5. 打后端健康检查。
6. 打前端首页探活。
7. 打关键公共接口。
8. 打关键鉴权接口。
9. 打开目标页面验证 live 数据。
10. 对比跨页面同一指标。
11. 跑相关测试。
12. 跑构建。

## 4. 本项目最实用的快速检查命令

### 4.1 服务验活

```powershell
curl.exe -fsS http://127.0.0.1:8000/health
curl.exe -I -s http://127.0.0.1:5173
```

### 4.2 端口与进程

```powershell
Get-Content .logs\backend.port, .logs\frontend.port

Get-NetTCPConnection -State Listen |
  Where-Object { $_.LocalPort -in @(5173,8000) } |
  Select-Object LocalAddress,LocalPort,OwningProcess,State
```

```powershell
Get-CimInstance Win32_Process |
  Where-Object { $_.ProcessId -in @(23132,31984) } |
  Select-Object ProcessId,ParentProcessId,Name,CommandLine
```

### 4.3 关键公共接口

```powershell
curl.exe -fsS "http://127.0.0.1:8000/api/v1/stats/public-goods-impact?range=month"
```

### 4.4 关键鉴权接口

```powershell
$body = @{ email = 'admin@foodbank.com'; password = 'admin123' } | ConvertTo-Json
$auth = Invoke-RestMethod -Uri 'http://127.0.0.1:8000/api/v1/auth/login' -Method Post -ContentType 'application/json' -Body $body
$headers = @{ Authorization = "Bearer $($auth.access_token)" }
Invoke-RestMethod -Uri 'http://127.0.0.1:8000/api/v1/stats/dashboard?range=month' -Headers $headers | ConvertTo-Json -Depth 8
```

## 5. 一页式最小检查清单

- [ ] 旧进程是否已清理。
- [ ] 前后端是否已干净重启。
- [ ] `.logs\backend.port` 是否存在且可信。
- [ ] `.logs\frontend.port` 是否存在且可信。
- [ ] 8000 是否返回健康检查成功。
- [ ] 5173 是否返回 200。
- [ ] 监听 8000 和 5173 的 PID 是否属于本项目。
- [ ] 目标页面是否真的读取 live 数据。
- [ ] 关键接口是否直接返回预期字段。
- [ ] 前后端字段名是否完全对齐。
- [ ] 同一指标在多个页面的口径是否一致。
- [ ] 当前 token 和角色是否正确。
- [ ] `.logs` 是否有第一现场错误信息。
- [ ] 相关测试是否跑过。
- [ ] 构建是否跑过。

## 6. 可直接复用的排错短语

- 先确认是不是旧进程问题。
- 先确认端口占用的是不是本项目。
- 先确认前后端是不是同一轮启动的。
- 先确认页面拿到的是 live 数据，不是占位数据。
- 先确认这个错误发生在启动层、接口层，还是渲染层。
- 先确认后端字段已经改完，前端映射有没有跟上。
- 先确认两个页面看的是不是同一个时间范围。
- 先确认两个页面对同一个指标用的是不是同一个定义。
- 先确认日志里有没有第一现场。
- 先确认是不是权限问题，而不是接口不存在。
- 先确认是不是 token 过期，而不是页面逻辑错误。
- 先确认是不是 mock 和真实接口混用了。
- 先确认是不是缓存了旧页面。
- 先确认是不是 dirty worktree 里的旧改动在干扰判断。
- 先做干净重启，再做细排查。

## 7. 适合作为结论句的表达

- 本轮问题首先不是代码逻辑错误，而是旧进程和端口状态混乱。
- 本轮问题首先不是接口不存在，而是页面没有吃到 live 数据。
- 本轮问题首先不是页面渲染异常，而是统计口径没有完全对齐。
- 本轮问题首先不是启动失败，而是前后端来自不同轮次的运行实例。
- 本轮问题首先不是 build 失败，而是运行时注入链断裂。
- 本轮问题首先不是数据为空，而是过滤条件或字段映射不一致。

## 8. 使用建议

- 适合在每次大联调前快速过一遍。
- 适合在发现“看起来都正常，但结果不对”时使用。
- 适合在写排查报告、答辩记录、周报总结时直接引用。
