## 1. 背景与问题定义

在本项目的前后端联调过程中，捐赠页和管理端数据仪表板都在展示“捐赠影响”相关指标，但两者存在一个典型的工程问题：

1. 两个页面虽然展示的是同一业务域的数据，但它们并不一定共享同一套统计逻辑。
2. 捐赠页属于 public 页面，管理端 dashboard 属于 admin 页面，权限、展示粒度、字段需求不同。
3. dashboard 页面并不是完全原生 React 页面，而是通过参考 HTML 嵌入到宿主页面中，再由宿主注入 live 数据服务。
4. 本地开发环境中经常存在旧进程残留、端口占用、前后端不是同一轮启动实例等问题，导致“页面看起来正常，但实际读的是旧服务”这种假象。

这类问题的难点不在于某一个接口报错，而在于：

- 页面可能能打开，但数据不是 live 的。
- 接口可能能返回，但前端字段没有完全对齐。
- 两个页面可能都有数据，但统计口径不一致。
- 本地服务可能都“活着”，但并不是当前代码启动出来的那一套。

因此，这轮工作的核心任务不是单点修 bug，而是完成一次“统计口径统一 + 页面接入真实数据 + 端到端验证 + 排错方法沉淀”的组合型改进。

## 2. 本轮工作概览

本轮工作可以概括为五个层面：

### 2.1 统一共享统计逻辑

将捐赠影响相关的核心指标计算抽取为后端共享服务，避免 public 页面和 admin 页面各算各的。

### 2.2 为 public 页面建立更合适的数据出口

新增面向捐赠页的精简公共接口，使 Donate Goods 页面不必依赖 admin dashboard 的大而全响应。

### 2.3 让页面真正接入 live 数据

Donate Goods 页面从静态占位指标切换为调用真实统计接口。

Admin Data Dashboard Preview 页面继续通过宿主注入 `dashboardDataService`，但移除了参考 HTML 中的 mock API 依赖，使其必须读取 live 数据。

### 2.4 建立回归验证

补充后端测试，显式验证：

- 新的 public goods impact 接口与 dashboard 使用的是同一套共享统计逻辑。
- 管理端 dashboard 的 donation KPI 与公共捐赠影响指标不会再次发生漂移。

### 2.5 沉淀排错方法论

在完成代码修复后，又额外整理了一份端到端检查与排错短句清单，目的是把这轮排查经验结构化，便于后续 AI 协作和 report 写作复用。

## 3. 本轮具体做了什么

## 3.1 提取共享统计服务，收敛“单一数据真相”

本轮新增了后端服务文件：

- `backend/app/services/impact_metrics_service.py`

该服务引入了 `SharedGoodsImpactSnapshot` 数据结构，用于统一承载以下统计结果：

- 当前周期物资捐赠件数
- 上一周期物资捐赠件数
- 本年累计物资捐赠件数
- 全部时间支持家庭数
- 当前周期支持家庭数
- 上一周期支持家庭数
- 合作机构数

这里的设计重点不是“多一个 service 文件”，而是建立明确的统计边界和业务定义。

具体实现上，这个服务统一处理了以下业务规则：

- 现金捐赠只统计 `completed` 状态。
- 实物捐赠只统计 `received` 状态。
- 家庭数按 `user_id` 去重，而不是按申请记录条数累计。
- 合作机构数按 donor identity 去重，优先使用 email，缺失时退化为 name。
- 日期判断兼容 `date` 与 `datetime`，并对时区信息做归一化处理。
- 周期判断统一通过 `current_start / next_start / previous_start` 完成。

这一步的意义非常大，因为它把“页面指标”转化成了“后端有明确定义的共享统计结果”。

换句话说，前端页面不再各自理解“什么叫物资捐赠件数”，而是共同依赖一套后端定义。

## 3.2 改造现有 public impact 统计逻辑，避免重复计算

在 `backend/app/routers/stats.py` 中，原本 public impact 相关逻辑里，家庭数和物资捐赠数是直接在路由内部遍历计算的。

本轮将这部分逻辑改为复用共享的 `calculate_shared_goods_impact_snapshot(...)`。

这样做有两个好处：

1. 旧有公共影响指标依然可用，不会破坏原有页面。
2. 物资相关的关键数字已经不再散落在不同路由里各自实现。

这是一种典型的“保持兼容的重构”方式。

它不是直接把原接口推翻重写，而是在不破坏现有功能的前提下，把逻辑慢慢收敛到共享服务上。

## 3.3 新增 `public-goods-impact` 公共接口

本轮新增接口：

- `GET /api/v1/stats/public-goods-impact`

这个接口的存在是一个很关键的技术决策。

如果直接让捐赠页去调用 admin dashboard 接口，会有几个问题：

- dashboard 接口需要 admin 权限，不适合 public 页面。
- dashboard 返回的数据体积更大，包含 donation、inventory、package、expiry、redemption 等多个板块。
- Donate Goods 页面只需要三张影响卡，直接拿 dashboard 全量响应不必要，也增加耦合。

因此，本轮没有简单复用现有 dashboard API，而是设计了一个“职责更清晰、权限更合适、负载更轻量”的公共接口。

这个接口只返回 Donate Goods 页面真正需要的三张卡：

- `goods_units_current_period`
- `families_supported`
- `partner_organizations`

从架构角度看，这是“共享统计逻辑 + 分场景输出接口”的设计。

也就是说：

- 核心统计逻辑共享。
- 输出接口按页面职责拆分。
- 公共页和管理页各自拿到更适合自己的响应结构。

这比“所有页面都去拿同一个大接口”更合理。

## 3.4 Donate Goods 页面从静态展示改为真实数据接入

在 `frontend/src/pages/DonateGoods/DonateGoods.tsx` 中，本轮完成了以下改动：

- 引入 `statsAPI.getPublicGoodsImpact()`。
- 新增 `PublicImpactMetric` 的前端类型使用。
- 将原本写死的 impact stats 占位值改为 `INITIAL_IMPACT_STATS`。
- 增加 `mapImpactStats()`，将接口返回的 `key-value-label-change` 结构映射到页面组件需要的展示卡片结构。
- 通过 `useEffect` 在页面加载时请求 live 数据。
- 补充 loading / ready / error 三种状态反馈。

这一步的意义在于：

1. 页面从静态演示状态切换为真实数据驱动状态。
2. 页面在接口失败时仍能优雅降级，而不是直接崩溃。
3. 页面展示文本由后端返回 label 决定，前端只负责映射，不再硬编码过多业务定义。

在 report 中，这可以被表述为：

“将展示型页面由静态示例数据改造为面向真实业务统计接口的数据驱动页面，并补充状态管理与错误回退逻辑。”

## 3.5 管理端 Dashboard 继续使用参考 HTML，但去除 mock 数据依赖

管理端 dashboard 的实现并不是一个完全原生的 React 图表页面，而是：

- 将 `scripts/Data_dashboard.html` 作为参考页面模板
- 通过 Vite 的虚拟模块读取这份 HTML
- 在 React 页面中以 iframe 的形式承载该参考页面
- 由宿主页面注入 `dashboardDataService` 和 `dashboardPostRender`

这种结构本身就带来一个难点：

页面可能“长得很像真实 dashboard”，但如果模板内部还有 mock API，那么它实际上展示的并不是项目真实数据。

本轮在 `scripts/Data_dashboard.html` 中做了一个重要改动：

- 移除了旧的 mock API 实现
- 明确要求通过 `window.dashboardDataService` 注入 live 数据
- 在没有注入 live service 时直接抛出错误，而不是悄悄回退到 mock 数据

这是一个非常有价值的工程改进。

因为它把“错误静默掩盖”改成了“错误显式暴露”。

具体来说：

- 以前：宿主注入失败时，页面仍可能显示 mock 图表，给人一种系统正常的假象。
- 现在：宿主注入失败时，问题会立刻暴露，便于尽早排查。

这符合真实系统联调的需求，也更适合项目答辩时说明“系统展示的就是实时业务数据，而非演示假数据”。

## 3.6 管理端 dashboard 宿主注入链继续保留，但增强兼容性

在 `frontend/src/pages/Admin/AdminDataDashboardPreview.tsx` 中，本轮没有推翻原有 iframe 注入架构，而是在保留现有方案的基础上做了增强：

- 继续由宿主通过 `dashboardDataService` 注入六类图表数据获取函数。
- 继续通过 `dashboardPostRender` 对卡片文本和表格内容做后渲染填充。
- 增加 `partner_organizations` 到 impact card 的键映射兼容。
- 使用 `Map<range, Promise<DashboardAnalyticsResponse>>` 做范围级缓存，避免同一时间范围下重复请求多次 dashboard 接口。

这是一种偏稳健的技术策略。

原因在于：

- 现有 dashboard HTML 模板已经较大，完全重写成本高、风险大。
- 通过宿主注入的方式可以保留参考页面的视觉结构，同时让数据接入真实接口。
- 局部增强兼容性比一次性推翻重写更适合当前项目阶段。

这类决策在 report 里可以描述为：

“在保持原有视觉模板资产可复用的前提下，通过宿主注入和接口适配完成真实数据接入，以降低重构风险并缩短联调周期。”

这能有效防止后续再次出现“一个页面改了，另一个页面没跟上”的回归问题。

## 4. 为什么做这样的技术决策

## 4.1 为什么要抽共享 service，而不是在两个接口里各自算一遍

原因很简单：

- 各算各的短期看更快，长期一定漂移。
- 同一业务指标如果出现两套定义，页面之间必然会出现不一致。
- 共享 service 更容易测试，也更容易在未来继续扩展。

这是一种“用少量结构化重构换长期一致性”的决策。

## 4.2 为什么要新建 `public-goods-impact`，而不是直接复用 dashboard 接口

这是出于职责分离、权限边界和接口负载控制的考虑。

dashboard 是 admin 视角的大盘接口，返回内容很多。

Donate Goods 页面只需要三张 impact 卡，直接复用 dashboard 接口会带来：

- 权限不合适
- 响应冗余
- 页面耦合 admin 结构
- 后续演进难度更大

因此，本轮采用了“共享逻辑复用、接口边界分离”的方式，而不是“所有页面都绑到一个大接口上”。

## 4.3 为什么保留 iframe + 参考 HTML + 宿主注入，而不是重写整个 dashboard

这是一个典型的阶段性工程决策。

完全重写 dashboard 的优点是结构统一，但缺点也非常明显：

- 重写成本高
- 风险大
- 当前模板和图表逻辑已有大量现成资产
- 短期目标是数据对齐，而不是重新设计整套 dashboard 架构

因此，本轮更合理的做法是：

- 保留现有参考 HTML 资产
- 去掉 mock 数据
- 强制宿主注入 live service
- 在宿主层完成接口接入和渲染补强

这是一个风险更低、收益更直接的策略。

## 4.4 为什么把 mock 回退去掉

因为 mock 回退在 demo 阶段方便，但在联调阶段有害。

联调阶段最怕的问题不是页面报错，而是“页面看起来没错，但其实展示的是假数据”。

因此，本轮选择让 dashboard 在没有 live service 时显式失败，而不是静默显示 mock 数据。

这样更利于暴露真实问题，也更符合教学项目和答辩演示的可信性要求。

## 4.5 为什么要把“清理旧进程”作为正式排查动作的一部分

因为在 Windows 本地开发环境里，残留 `node` 和 `python` 进程是非常典型的干扰因素。

如果不先确认：

- 端口到底被谁占用
- 当前页面到底连的是哪一套服务
- 监听 8000/5173 的 PID 是否来自本仓库

那么很多后续判断都会建立在错误前提上。

所以本轮把“进程与端口核对”视作端到端验证的第一步，而不是一个偶发的辅助操作。

## 5.5 验证方式覆盖了单元、构建和真实运行时

## 6. 本轮遇到的难点

## 6.1 难点一：同一业务域，不同页面职责，不同接口形态

捐赠页和 dashboard 都属于捐赠影响统计，但：

- 一个是 public 页面
- 一个是 admin 页面
- 一个只需要 3 个卡片指标
- 一个需要完整 analytics 大盘

如果只从“复用”角度思考，容易把二者硬绑到一个接口上；
如果只从“页面独立”角度思考，又会造成统计逻辑重复。

本轮的解决方式是：

“共享逻辑，不共享不必要的接口边界。”

这实际上是一个比简单代码修改更高层的架构判断。

## 6.2 难点二：dashboard 并非纯 React 页面

管理端 dashboard 通过 iframe 承载参考 HTML，并由宿主进行数据注入。这意味着排查时必须同时考虑：

- 模板 HTML 本身
- 宿主 React 页面
- 接口封装层
- 运行时注入链

这类问题往往不是单文件就能看出来，而是跨层排查。

本轮通过去掉 mock fallback，并保留宿主注入机制，使问题边界更清晰，降低了后续排错难度。

## 6.3 难点三：本地旧进程会制造大量假象

本轮真实排查时，曾发现多个 `node`、`python` 进程同时存在。表面上 5173 和 8000 都可访问，但如果不确认 PID 和命令行，很容易误判当前页面实际连到的是哪一套服务。

这类问题的困难在于：

- 页面可能正常打开
- 接口可能也有返回
- 但返回的并不是当前代码版本对应的服务

这使得“看起来没问题”反而成为一种干扰。

因此，本轮把“停止旧服务 -> 干净重启 -> 复核 PID -> 再做接口核对”固定成正式方法。

## 6.4 难点四：统计口径问题比语法错误更隐蔽

语法错误、接口 500、构建失败通常都比较容易发现。

真正棘手的是下面这种问题：

- 页面 A 和页面 B 都有值
- 页面 A 和页面 B 都没有报错
- 但同名指标实际上不是同一套定义

例如：

- 一个算的是当前周期已接收的物资数量
- 另一个算的是全年总数
- 一个家庭按申请条数累加
- 另一个按用户去重

这种问题如果没有共享 service 和回归测试，后续非常容易再次出现。

## 7. 这么做带来的好处

## 7.1 业务口径更一致

同一类物资捐赠影响指标不再散落在多个路由中独立实现，降低了数据漂移风险。

## 7.2 页面可信度更高

Donate Goods 页面不再显示静态示例数字，dashboard 也不再隐式回退 mock 数据，页面展示更接近真实系统状态。

## 7.3 维护成本更低

未来如果需要调整“合作机构数”或“家庭支持数”的定义，只需要先改共享统计服务，再验证各出口接口是否符合需求，而不必满项目找重复逻辑。

## 7.4 测试更有针对性

本轮补的测试不是普通的字段测试，而是跨接口一致性测试，更能防止“局部修改造成全局漂移”。

## 7.5 排错效率更高

通过补充排错文档和固定检查顺序，后续遇到类似问题时，能更快判断是：

- 旧进程问题
- 注入链问题
- 权限问题
- 字段映射问题
- 统计口径问题

## 7.6 对 report 和答辩更友好

这一轮工作的叙述价值也比较高，因为它不只是“修了一个 bug”，而是可以清楚展示：

- 如何发现问题
- 如何拆解问题
- 如何做架构取舍
- 如何做验证闭环
- 如何把经验沉淀为方法文档

这类内容在课程 report 和毕业答辩中通常比单纯的界面优化更有说服力。

## 9. 从 report 写作角度可以怎么表述

下面这些表述可以直接作为后续 report 的素材基础：

### 9.1 从“问题导向”角度写

本轮工作的核心问题并不是页面崩溃，而是同一业务指标在不同页面之间存在潜在口径漂移风险，同时本地联调环境中旧进程与端口残留容易制造“假正常”现象，因此需要同时完成数据逻辑统一与运行环境排错机制完善。

### 9.2 从“架构改进”角度写

为了解决跨页面统计逻辑重复的问题，本轮将物资捐赠影响指标提取为后端共享服务，并在此基础上分别为公共页面和管理页面提供职责更清晰的数据出口，实现“共享核心逻辑、分场景接口输出”的架构改进。

### 9.3 从“工程取舍”角度写

考虑到现有 dashboard 已具备较成熟的参考页面结构，本轮并未选择高风险的全量重写，而是保留参考 HTML 与宿主注入架构，通过去除 mock 数据依赖和接入 live service 的方式，在较低改动成本下实现真实数据联通。

### 9.4 从“验证闭环”角度写

本轮采用了“共享逻辑重构 + 自动化测试 + 真实服务验活 + 端口与进程核对”的组合验证策略，确保修复结果不仅在单元测试中成立，也在本地真实运行环境中有效。

### 9.5 从“项目亮点”角度写

本轮工作的亮点在于同时兼顾了业务口径一致性、接口边界合理性、现有资产复用、运行时可信度和排错方法沉淀，使其既是一次功能修复，也是一次具有工程方法论价值的改进。

## 11. 结论

总体来看，这一轮工作的价值不只是“让捐赠页显示了正确数字”，而是在更高层面完成了以下事情：

- 把分散的统计逻辑收敛成共享服务。
- 把 public 页面和 admin 页面连接到更合理的接口边界。
- 把 dashboard 从“可能显示 mock 数据的演示页面”推进到“必须依赖 live 数据的真实页面”。
- 把联调过程中的经验整理成可重复执行的排错方法。
- 把技术修复结果通过测试、构建、服务验活和进程核对形成了完整闭环。

因此，这一轮更适合被定义为一次“数据一致性与联调可靠性提升”的工程改进，而不只是一次局部页面修复。
