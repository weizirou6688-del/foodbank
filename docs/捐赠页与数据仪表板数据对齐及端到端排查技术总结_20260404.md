# 捐赠页与数据仪表板数据对齐及端到端排查技术总结

日期：2026-04-04

## 1. 适用范围

本文记录以下模块在一次联调中的数据对齐与排查结论：

- 公开捐赠页 `frontend/src/pages/DonateGoods/DonateGoods.tsx`
- 后台统计接口 `backend/app/routers/stats.py`
- 统计聚合服务 `backend/app/services/impact_metrics_service.py`
- 后台仪表板预览页 `frontend/src/pages/Admin/AdminDataDashboardPreview.tsx`
- 嵌入式仪表板页面 `scripts/Data_dashboard.html`

## 2. 主要问题现象

- 公开捐赠页展示的 impact 数据与后台仪表板数值不一致。
- 页面可以正常渲染，但部分模块仍可能落回默认值或 mock 数据。
- 同一时间范围下，public 与 admin 的统计口径不完全一致。
- 本地联调时，旧进程、旧端口、旧页面缓存会放大定位难度。

## 3. 根因归纳

### 3.1 统计逻辑分散

公开页、后台仪表板、HTML 预览页并不总是使用同一套统计聚合结果。只要其中一处仍走旧逻辑，就会出现同页不同数、前后台不同数的问题。

### 3.2 回退逻辑过宽

当 live 请求失败、注入链未完成，或 iframe 内脚本未成功执行时，页面可能静默回退到默认展示。这种处理可以保页面不报错，但会掩盖真实的数据链路问题。

### 3.3 数据范围不一致

以下口径只要任意一项不一致，就会导致结果偏差：

- 时间范围
- `food_bank_id` 过滤条件
- donation 状态筛选
- `date` 与 `datetime` 的边界处理
- public 指标与 admin 指标的聚合粒度

### 3.4 仪表板是两段式注入链

后台预览并不是单一 React 组件直接渲染所有数据，而是存在以下链路：

1. React 容器页加载预览壳层。
2. 预览壳层承载 HTML 仪表板。
3. HTML 仪表板通过 `window.dashboardDataService` 获取数据。
4. 页面完成渲染后再通过 `window.dashboardPostRender` 做二次处理。

任意一段未完成，都会表现为旧数据、空数据或默认数据。

### 3.5 运行实例不一致

前端、后端、脚本页若来自不同轮次的启动实例，表面上页面依然可打开，但实际请求和展示已经不在同一套运行上下文中。

## 4. 对齐原则

- public 与 admin 共用同一份后端统计结果。
- 预览页与 live 页面优先读取真实服务数据。
- fallback 只用于显式降级，不用于静默掩盖问题。
- 排查顺序先看进程、端口、接口，再看页面和样式。

## 5. 关键代码落点

### 5.1 后端统计聚合

`backend/app/services/impact_metrics_service.py` 中的以下对象是对齐核心：

- `GoodsImpactSnapshot`
- `calculate_goods_impact_snapshot(...)`

这层负责把 goods donation 相关统计集中到同一份快照结构中，供不同接口复用。

### 5.2 统计接口出口

`backend/app/routers/stats.py` 负责向前端提供 public 和 admin 侧统计数据，其中包括：

- `get_public_goods_impact_metrics(...)`
- 仪表板相关汇总接口

如果接口层追加了不同过滤条件，即使底层服务一致，最终展示仍会偏差。

### 5.3 公开捐赠页

`frontend/src/pages/DonateGoods/DonateGoods.tsx` 中的 `INITIAL_IMPACT_STATS` 仅适合作为初始展示或失败兜底，不能替代 live 数据结果。

### 5.4 后台仪表板预览

`frontend/src/pages/Admin/AdminDataDashboardPreview.tsx` 负责承载后台预览逻辑，需要重点确认：

- iframe 是否成功加载
- 注入的数据服务是否已挂载
- 预览页使用的是 live 数据还是回退数据

### 5.5 HTML 仪表板页

`scripts/Data_dashboard.html` 需要重点确认以下两个入口是否可用：

- `window.dashboardDataService`
- `window.dashboardPostRender`

## 6. 推荐排查顺序

1. 先确认当前前后端进程、端口和启动时间是否一致。
2. 直接调用 live 统计接口，记录返回值。
3. 对比 public 与 admin 接口的过滤条件和时间范围。
4. 在浏览器 Network 与 Console 中确认是否发生了 fallback。
5. 检查 iframe、HTML 页和数据注入链是否完整。
6. 回归验证公开捐赠页、后台预览页和后台仪表板是否一致。

## 7. 验收口径

满足以下条件时，可认为本轮对齐有效：

- 同一时间范围下，public 与 admin 的 goods impact 指标一致。
- 预览页和 live 页面不再优先显示旧 mock 数据。
- live 数据失败时，页面明确显示降级或错误状态。
- 相关统计测试或等价人工核对能够复现一致结果。

## 9. 结论

本问题的核心不在单个页面，而在统计口径、数据出口和注入链的一致性。当前实现已经改为 bank-scoped 统计快照与统一接口过滤条件，并继续收紧 legacy fallback 的使用范围，这样公开页与后台仪表板的数据才能保持稳定对齐。
