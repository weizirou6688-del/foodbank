# 数据库落地过程阻塞原因与改进记录

日期：2026-03-30

## 1. 目的

本记录用于说明本次本地 PostgreSQL 落地、数据库修复与迁移过程中导致进度拖延的真实原因，并给出后续避免重复阻塞的处理方案。

## 2. 本次实际出现的阻塞原因

### 2.1 项目文档缺少“项目数据库账号”和“PostgreSQL 超级用户密码”的明确区分

现象：

- README 中写了 `admin123`、`supermarket123` 等业务登录密码。
- 但这些密码容易被误认为是 PostgreSQL 管理员密码。
- 项目数据库账号 `foodbank / foodbank` 的用途没有在 README 首页明确写清。

影响：

- 在连接数据库时容易混淆“应用登录密码”和“数据库管理密码”。
- 排障时先后尝试了错误的密码路径，浪费了时间。

改进措施：

- 已在 [README.md](c:/Users/weicq/Desktop/foodbank/README.md) 中补充项目数据库账号说明。
- 明确标注：`foodbank / foodbank` 是项目数据库账号，不是 PostgreSQL 超级用户密码。

### 2.2 本机 PostgreSQL 已安装，但项目约定账号未初始化或密码未对齐

现象：

- PostgreSQL 服务实际已经安装并运行。
- `5432` 端口正常监听。
- 但项目默认账号 `foodbank / foodbank` 最初无法登录。

影响：

- 后端无法使用项目默认连接串连通数据库。
- RLS、迁移、后端接口联调都被阻塞。

改进措施：

- 已创建并验证项目数据库账号 `foodbank / foodbank`。
- 已确认 `foodbank` 数据库可用。

### 2.3 Windows 服务控制需要管理员权限，代理环境无法直接代替用户完成

现象：

- 即使在代理流程中申请了提权，执行环境仍无法真正控制 Windows 服务。
- `Restart-Service postgresql-x64-17` 与 `sc.exe stop/start` 都出现 `Access is denied` 或 `Cannot open service`。

影响：

- 每次临时修改 `pg_hba.conf` 后，都必须由用户手动以管理员身份重启 PostgreSQL 服务。
- 这导致自动化流程被迫中断等待人工配合。

改进措施：

- 后续文档里明确写出：凡是修改 `pg_hba.conf`，必须提前准备管理员 PowerShell。
- 优先避免再次依赖这类临时认证切换。

### 2.4 PostgreSQL 认证方式为 `scram-sha-256`，而超级用户密码未记录

现象：

- 本机 `pg_hba.conf` 使用 `scram-sha-256`。
- 仓库中没有保存 PostgreSQL 超级用户 `postgres` 的真实密码。
- 无法直接用超级用户密码完成数据库修复。

影响：

- 排障时只能通过临时切换本地认证到 `trust` 的方式进入数据库。
- 这增加了操作步骤和协作成本。

改进措施：

- 后续默认不依赖超级用户密码参与日常项目开发。
- 项目统一只依赖 `foodbank / foodbank` 作为开发数据库账号。

### 2.5 数据库处于“半初始化状态”，导致 Alembic 无法直接升级

现象：

- `foodbank` 数据库内已有残留 `users` 表。
- 但数据库中没有 `alembic_version` 表。
- Alembic 认为这是一个空库，于是在执行初始迁移时与现有表冲突。

影响：

- `alembic upgrade head` 首次执行报错：`users already exists`。
- 这类问题如果不识别清楚，很容易误判为迁移脚本损坏。

改进措施：

- 已清空旧的 `public schema` 并重新执行全量迁移。
- 当前 Alembic 版本已到 `20260326_0013`。
- 后续建议：开发环境不要混用 `create_all` 残留结构与 Alembic 全量迁移。

### 2.6 从错误工作目录读取配置，触发了与项目无关的环境变量污染

现象：

- 某些验证命令在仓库根目录运行，而不是在 `backend` 目录运行。
- `Settings()` 在加载配置时命中了其他环境变量，出现：
  - `secret_key missing`
  - `debug=release` 无法解析

影响：

- 看起来像“项目配置有问题”，但其实是执行目录和环境变量污染问题。
- 额外增加了误判成本。

改进措施：

- 所有读取后端配置、运行 Alembic、运行后端命令，统一在 `backend` 目录下执行。
- 后续验证时优先复用 `.env` 所在目录作为工作目录。

## 4. 后续避免再次拖延的具体措施

### 4.1 文档层

要做的事：

1. 在 README 中固定写清项目数据库账号。
2. 在 Windows 数据库指南中加入“常见阻塞原因”部分。
3. 把“应用登录密码”和“数据库密码”分开写，避免混淆。

### 4.2 环境层

要做的事：

1. 保持 `foodbank / foodbank` 作为开发数据库统一凭证。
2. 不再依赖记忆 PostgreSQL 超级用户密码开展日常开发。
3. 后续如非必要，不再修改 `pg_hba.conf`。

### 4.3 执行流程层

要做的事：

1. 所有 Alembic 操作固定从 `backend` 目录执行。
2. 新机器首次启动时先验证数据库连接，再启动后端。
3. 如果数据库已经不是空库，先检查是否存在 `alembic_version`，避免迁移撞表。

### 4.4 项目结构层

要做的事：

1. 逐步减少开发环境对 `Base.metadata.create_all` 的依赖。
2. 让 Alembic 成为唯一可信的数据库结构来源。
3. 在需要大量分析数据时，优先使用项目内合成数据脚本，而不是手工造表或混入脏数据。

## 5. 可以直接写进 report 的总结表述

本次数据库落地过程的主要阻塞并非来自业务代码逻辑错误，而是来自开发环境初始化不一致，包括数据库账号约定未显式记录、Windows 服务重启权限不足、PostgreSQL 超级用户密码缺失、旧表结构残留与 Alembic 版本记录缺失等。经过修复后，项目数据库已被统一到标准状态，后续开发将以 `foodbank / foodbank` 作为本地开发数据库凭证，并以 Alembic 迁移作为唯一数据库结构基线，以降低后续环境排障成本。
