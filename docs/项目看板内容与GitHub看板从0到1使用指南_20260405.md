# 项目看板内容与 GitHub 看板从 0 到 1 使用指南_20260405

## 1. 先给结论

基于当前项目代码与文档状态（2026-04-05），你的 `foodbank` 项目已经不是“从零开始”的空项目，而是一个已经有主链路、后台管理、捐赠、申请、库存、补货、统计与演示材料基础的课程项目。

你的截图里把以下几类内容放到了同一块板上：

- 流程状态：`Backlog` / `Sprint Backlog` / `In progress` / `In review` / `Done`
- 历史问题：`Issues--First Sprint` / `Issues--Second Sprint` / `Issues--3rd Sprint`
- 文档类工作：`Documentation-Keep in Progress`
- 答辩类工作：`Presentation`

这不是不能用，但它不是一个纯“开发状态板”，更像一个“课程项目总控板”。

最稳妥的做法是：

1. 建一个 `Portfolio Board`，列名完全按你截图来，用于老师/组员一眼看全局。
2. 再建一个 `Execution Board`，只保留 `Backlog / Sprint Backlog / In progress / In review / Done`，用于日常推进。
3. `Roadmap` 单独建成 roadmap view，不要硬塞进 board。

如果你时间紧，只建一个板也可以，直接照本文档里的 `Board Section` 建列即可。

## 3. 推荐你在 GitHub Projects 里建的字段

如果你是 GitHub 新手，先不要追求复杂自动化，先把字段建对。

### 3.1 必建字段

- `Board Section`
  - 类型：Single select
  - 用途：专门复刻你截图里的列标题
  - 选项建议：
    - `Backlog`
    - `Sprint Backlog`
    - `In progress`
    - `In review`
    - `Done`
    - `Issues--First Sprint`
    - `Issues--Second Sprint`
    - `Issues--3rd Sprint`
    - `Documentation-Keep in Progress`
    - `Presentation`
- `Status`
  - 类型：Single select
  - 选项建议：
    - `Todo`
    - `In progress`
    - `In review`
    - `Done`
- `Priority`
  - 类型：Single select
  - 选项建议：
    - `P0`
    - `P1`
    - `P2`
    - `P3`
- `Estimate`
  - 类型：Number
  - 建议值：
    - `1`
    - `2`
    - `3`
    - `5`
    - `8`
- `Area`
  - 类型：Single select
  - 选项建议：
    - `Frontend`
    - `Backend`
    - `Fullstack`
    - `Testing`
    - `Docs`
    - `Presentation`
- `Sprint`
  - 类型：Single select 或 Iteration
  - 如果你不会用 iteration，先用 single select：
    - `Sprint 1`
    - `Sprint 2`
    - `Sprint 3`
    - `Final Polish`
- `Target Date`
  - 类型：Date

### 3.2 新手最容易踩的坑

- 不要把 `Presentation`、`Documentation`、`Issues--First Sprint` 这些内容直接当成 `Status`。
- 更好的做法是：
  - 用 `Board Section` 负责截图里的列
  - 用 `Status` 负责真正的开发流程

### 3.3 如果你用的是公开项目

GitHub 的 `issue fields` 目前有可见性和项目类型限制。如果你完全不会区分 `issue fields` 和 `project custom fields`，直接用项目里的普通自定义字段就够了，不要一开始就搞组织级字段。

## 16. 如果你完全不会用 GitHub 看板，按这个步骤做

下面按“从 0 到 1”写，尽量不假设你有经验。

### 第一步：新建一个 Project

如果你是仓库拥有者或组织成员：

1. 打开 GitHub 仓库。
2. 进入 `Projects`。
3. 点击 `New project`。
4. 先选空白项目即可，不用纠结模板。
5. 名称建议：
   - `Foodbank Project Board`

### 第二步：先建字段，不要先急着建很多卡

先把这几个字段建好：

- `Board Section`
- `Status`
- `Priority`
- `Estimate`
- `Area`
- `Sprint`
- `Target Date`

如果只想最简化，最少也要建：

- `Board Section`
- `Priority`
- `Estimate`

### 第三步：创建一个和截图一样的板

1. 新建一个 view。
2. Layout 选 `Board`。
3. 这个 view 命名为：
   - `Portfolio Board`
4. 在 board 设置里把 `Column field` 设为：
   - `Board Section`
5. 把列值依次加成：
   - `Backlog`
   - `Sprint Backlog`
   - `In progress`
   - `In review`
   - `Done`
   - `Issues--First Sprint`
   - `Issues--Second Sprint`
   - `Issues--3rd Sprint`
   - `Documentation-Keep in Progress`
   - `Presentation`

### 第四步：再建一个真正用来推进开发的板

1. 再新建一个 view。
2. Layout 也选 `Board`。
3. 命名为：
   - `Execution Board`
4. `Column field` 选：
   - `Status`
5. 列值：
   - `Todo`
   - `In progress`
   - `In review`
   - `Done`

这个板才是你平时移动卡片最频繁的地方。

### 第五步：开始录入卡片

你有两种方式：

- 方式 A：先建 `Draft issue`
  - 适合还没完全确定细节的时候
- 方式 B：直接建仓库 issue
  - 适合准备认真追踪、关联 PR 和提交记录的时候

如果你是新手，我建议：

1. 先把本文件第 6 到第 15 节的标题先作为 `Draft issue` 录进去。
2. 其中真正要做的几张卡，再转成正式 issue。

### 第六步：每张卡至少填这几个值

- `Board Section`
- `Status`
- `Priority`
- `Estimate`
- `Area`
- `Sprint`

例如：

- `FE-02 收口 Admin dashboard 预览页与 live 数据接入细节`
  - Board Section：`Sprint Backlog`
  - Status：`Todo`
  - Priority：`P0`
  - Estimate：`3`
  - Area：`Frontend`
  - Sprint：`Sprint 3`

### 第八步：每天怎么移动卡片

最简单的规则：

- 还没开始：放 `Sprint Backlog` 或 `Todo`
- 开始做了：移到 `In progress`
- 自己做完，准备检查：移到 `In review`
- 确认通过：移到 `Done`

如果你在用截图同款总控板：

- 那么同步更新 `Board Section`
- 但真正反映开发进度的，还是 `Status`

### 第九步：每周怎么维护

每周只做 4 件事：

1. 从 `Backlog` 里挑 3 到 6 张最重要的卡放到 `Sprint Backlog`
2. 给每张卡设 `Priority` 和 `Estimate`
3. 做完就移动 `Status`
4. 把本周遇到的新问题记到对应 `Issues--Sprint X` 列

### 第十步：老师最容易看懂的用法

如果你是做课程项目展示，我建议你展示时按这个顺序给老师看：

1. `Portfolio Board`
   - 说明你不仅在做代码，还在管理问题、文档和展示材料
2. `Execution Board`
   - 说明你知道任务推进状态
3. `Roadmap`
   - 说明你知道项目时间安排
4. `Done`
   - 对应到项目里已经完成的真实功能和真实文档
