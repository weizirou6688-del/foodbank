# Windows 本地数据库与分析数据生成指南

日期：2026-03-29

## 1. 目标

这份指南解决两个问题：

1. 在 Windows 本机落地 PostgreSQL，让项目真正拥有可运行的本地数据库。
2. 在真实业务数据不足时，生成一批“可用于统计分析与 report 截图”的合成数据。

## 3. 资源占用结论

对你这个课程项目来说，本地 PostgreSQL 的资源占用通常是可接受的。

### 3.1 默认 demo 数据很小

项目启动后自动补的数据只有少量基础记录，例如：

- 3 个 demo 用户
- 3 个 food bank
- 少量库存、食品包和批次数据

这部分几乎不会给内存带来压力。

### 3.2 合成分析数据也可控

新加的数据生成脚本支持按天数和数量控制规模。即使你生成：

- 180 天
- 每天 3 笔现金捐赠
- 每天 2 笔实物捐赠
- 每周 14 条申请记录

总量也只是几千条级别，远远不是“几亿行”。

### 3.3 占用的主要是磁盘，不是内存

数据库表中的记录主要占硬盘。只有在你查询时，部分数据才会进入内存。对这种中小型课程项目，本地同步一份是合理的。

## 4. 外部 food bank feed 要不要存本地

建议区分两类数据：

### 4.1 外部 feed

适合用作：

- 地图找 food bank
- 第三方机构列表展示
- 作为外部参考数据源

### 4.2 本地业务库

适合保存：

- 用户
- 申请
- 捐赠
- 库存
- 补货
- 统计分析结果所依赖的数据

### 4.3 推荐方案

课程项目最实用的方案是：

1. 本地 PostgreSQL 保存你自己的业务数据。
2. 外部 food bank feed 作为补充来源。
3. 如果后面确实要做地区统计，再把 feed 的必要字段同步进本地，而不是做完整镜像。

## 5. Windows 本机落地 PostgreSQL 的最小方案

### 5.1 安装 PostgreSQL

安装 PostgreSQL 15 或 16 均可。安装完成后确认：

- PostgreSQL 服务已启动
- `5432` 端口可用
- 你记得超级用户密码

### 5.2 创建数据库和用户

推荐在 `psql` 中执行：

```sql
CREATE USER foodbank WITH PASSWORD 'foodbank';
CREATE DATABASE foodbank OWNER foodbank;
GRANT ALL PRIVILEGES ON DATABASE foodbank TO foodbank;
```

如果用户或数据库已存在，就跳过重复创建。

### 5.3 检查项目配置

确认 [backend/.env](c:/Users/weicq/Desktop/foodbank/backend/.env) 中的连接串是：

```env
DATABASE_URL=postgresql+asyncpg://foodbank:foodbank@localhost:5432/foodbank
```

### 5.4 执行迁移

在项目后端目录执行：

```powershell
cd c:\Users\weicq\Desktop\foodbank\backend
alembic upgrade head
```

### 5.5 启动后端，让 demo 数据自动补齐

```powershell
cd c:\Users\weicq\Desktop\foodbank\backend
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000
```

启动时会自动执行 demo 数据补齐逻辑。

## 6. 如何生成用于统计分析的更多数据

### 6.1 直接使用批处理脚本

在仓库根目录执行：

```powershell
scripts\generate_analytics_data.bat
```

默认会生成 180 天范围内的分析用数据。

### 6.2 指定生成天数

例如生成 90 天数据：

```powershell
scripts\generate_analytics_data.bat 90
```

### 6.3 直接运行 Python 脚本并自定义规模

```powershell
cd c:\Users\weicq\Desktop\foodbank\backend
python scripts\generate_analytics_data.py --days 180 --public-users 24 --cash-per-day 3 --goods-per-day 2 --applications-per-week 14
```

可调参数说明：

- `--days`：生成多少天的数据
- `--public-users`：创建多少个分析用 public 用户
- `--cash-per-day`：平均每天生成多少现金捐赠
- `--goods-per-day`：平均每天生成多少实物捐赠
- `--applications-per-week`：平均每周生成多少申请记录

## 7. 推荐的数据规模

为了让图表更好看、统计更稳定，又不让本机负担过重，建议你先用下面这组：

```powershell
python scripts\generate_analytics_data.py --days 180 --public-users 30 --cash-per-day 4 --goods-per-day 3 --applications-per-week 18
```

这个规模通常就足够你做：

- donation trend
- weekly donation summary
- package popularity
- stock gap / restock pressure
- application activity report

## 8. 如果你担心机器资源不够

那就从更小的规模开始：

```powershell
python scripts\generate_analytics_data.py --days 60 --public-users 12 --cash-per-day 2 --goods-per-day 1 --applications-per-week 8
```

如果这组运行和查询都很顺，再逐步增大。

## 9. 适合写进 report 的解释

如果老师或评审问你“为什么没有直接使用海量真实数据”，你可以这样表述：

1. 本项目重点在于验证食品银行业务流程、前后端联调与数据库建模，而不是处理超大规模数据仓库场景。
2. 真实外部数据源主要作为 food bank 列表补充来源，不直接覆盖项目内部业务表。
3. 为保证本地开发可复现性和统计分析可展示性，项目采用了“真实结构 + 可控规模的合成数据生成”方案。
4. 该方案既满足功能验证与报表展示，又避免对个人开发设备造成不必要的资源压力。

## 10. 下一步建议

最推荐的顺序是：

1. 先安装并启动本地 PostgreSQL。
2. 执行 Alembic 迁移。
3. 启动后端，确认 demo 数据自动生成。
4. 运行合成数据脚本扩充统计样本。
5. 再去验证统计页面、导出图表和撰写 report。

## 11. 本次实际阻塞记录

为了避免后续重复踩坑，本次本机数据库落地过程中出现的真实阻塞点如下：

1. README 里原先只有应用登录密码，没有在首页明确区分项目数据库账号与 PostgreSQL 超级用户密码。
2. 本机 PostgreSQL 服务虽然已安装，但项目数据库账号 `foodbank / foodbank` 最初并未处于可直接登录状态。
3. Windows 服务重启需要管理员权限，导致修改 `pg_hba.conf` 后不能完全自动完成重载。
4. `foodbank` 数据库在迁移前不是空库，存在残留 `users` 表，但没有 `alembic_version`，从而引发 Alembic 撞表。
5. 部分验证命令如果不在 `backend` 目录运行，会被其他环境变量污染，触发假性配置报错。

对应改进原则：

1. 后续本地开发统一使用 `foodbank / foodbank` 作为项目数据库账号。
2. 所有 Alembic 与后端配置命令都固定在 `backend` 目录执行。
3. 遇到旧库时先检查 `alembic_version`，不要直接假设它是空库。
4. 如非必要，不再通过修改 `pg_hba.conf` 处理常规开发问题。
5. 更完整的阻塞原因与改进方案见：[数据库落地过程阻塞原因与改进记录_20260330.md](c:/Users/weicq/Desktop/foodbank/docs/数据库落地过程阻塞原因与改进记录_20260330.md)。
